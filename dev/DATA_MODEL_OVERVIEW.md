# Data Model Overview

← Back to Developer Documentation (`/dev/README.md`)

---

## Overview

The QuestForge Campaign Distillery is built on a **composable data model**.

Campaigns are generated by combining a small set of distinct components:

* Core → meaning
* System → play experience
* Tone → emotional delivery
* Genre → narrative framing
* Environment → physical context

Each layer has a **specific role** and must remain cleanly separated.

---

## Core Principle

> Each layer answers a different question.

Mixing responsibilities between layers leads to:

* redundant entries
* unclear output
* reduced system flexibility

---

## Layer Breakdown

---

## Core (Meaning)

**Question:**

> What is this campaign fundamentally about?

Core defines the **central tension, theme, or narrative idea**.

---

### Examples

* hidden truth
* power has a cost
* survival against overwhelming force
* war of ideologies

---

### What Core IS

* a narrative tension
* a thematic driver
* a source of meaning

---

### What Core is NOT

* a gameplay mechanic
* a setting
* a tone
* a combination of ideas

---

### Test

If it answers:

> “What is this story really about?”

→ it belongs in Core

---

## System (Play Experience)

**Question:**

> What does play feel like moment-to-moment?

System defines the **player interaction loop**.

---

### Examples

* clue web
* resource scarcity
* faction reputation
* escalation pressure

---

### What System IS

* a gameplay loop
* a player-facing mechanic
* a recurring decision pattern

---

### What System is NOT

* a theme
* a setting
* an abstract design concept
* a vague idea like “engagement”

---

### Test

If it answers:

> “What are players doing repeatedly?”

→ it belongs in System

---

## Tone (Emotional Delivery)

**Question:**

> How does this campaign feel emotionally?

Tone defines the **emotional lens** applied to everything else.

---

### Examples

* heroic
* grimdark
* psychological
* lighthearted
* mythic

---

### What Tone IS

* emotional direction
* narrative intensity
* stylistic feel

---

### What Tone is NOT

* a setting
* a mechanic
* a theme

---

### Test

If it answers:

> “How does this feel?”

→ it belongs in Tone

---

## Genre (Narrative Framing)

**Question:**

> What kind of story space is this?

Genre defines **story expectations and narrative conventions**.

---

### Examples

* gothic
* classic fantasy
* cosmic eldritch
* noir

---

### What Genre IS

* narrative style
* storytelling conventions
* aesthetic expectations

---

### What Genre is NOT

* a specific environment
* a gameplay system
* a pure tone

---

### Note (Important)

The current genre layer is **overloaded** and may be split in future versions into:

* era / timeline
* aesthetic
* world condition

Do not expand genre aggressively until this refactor is complete.

---

## Environment (Physical Context)

---

* open ocean

---

### What Environment IS

* location
* physical space
* spatial constraints

---

### What Environment is NOT

* a tone
* a theme
* a gameplay system

---

### Test

If it answers:

> “Where does this happen?”

→ it belongs in Environment

---

## Composition Model

Campaigns are built by combining layers:

```text
Core + System + Tone + Genre + Environment
```

Each layer contributes something different:

* Core → meaning
* System → behavior
* Tone → emotional feel
* Genre → narrative expectations
* Environment → physical context

---

## Common Mistakes

---

### ❌ Combined Entries

Example:

* “frontier horror”

Problem:

* mixes environment + tone + genre

Solution:

* use composition instead

---

### ❌ Overly Abstract Concepts

Example:

* “narrative progression”
* “player engagement”

Problem:

* not player-facing
* not usable in output

---

### ❌ Redundant Entries

Example:

* “hidden conspiracy” vs “hidden truth”

Problem:

* duplicates meaning

Solution:

* refine existing entry

---

### ❌ Misclassified Entries

Example:

* putting “resource scarcity” in Core instead of System

Problem:

* breaks composability

---

## Expansion Guidelines (Summary)

When adding new entries:

* must be **specific**
* must be **portable**
* must produce **distinct output**

If it can be built from existing layers:
→ do not add it

---

## Future Evolution

The model may evolve, but must remain:

* composable
* clearly separated
* easy to reason about

Major structural changes (e.g. genre refactor) should be done carefully and not during active expansion phases.

---

## Summary

The system works because:

* each layer has a clear role
* layers do not overlap
* campaigns are built through composition

Maintain these boundaries to preserve flexibility and scalability.
